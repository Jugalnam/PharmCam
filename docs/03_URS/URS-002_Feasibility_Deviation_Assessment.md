# URS-002 — URS Feasibility & Deviation Assessment (URS 적합성·일탈 검토)

**시스템명:** PharmCam — 검체·시료 사진 기록 시스템
**문서 ID:** URS-002
**버전:** 1.0 (Approved)
**작성일:** 2026-06-05
**확정일:** 2026-06-05
**상위 문서:** URS-001 (User Requirement Specification)

> **이 문서가 하는 일:** DQ(설계)에 들어가기 전, **우리가 택한 구현 방식으로 URS 42개를 모두 만족할 수 있는지** 검토한다.
> 소프트웨어만으로 완전히 충족되지 않는 요구사항은 **일탈(deviation)**로 정식 기록하고, 사유·위험·처리방안을 명시한다.
> 미해결 일탈이 남으면 DQ로 넘어가지 않는다(VP §7 일탈 처리 원칙).

---

## 1. 전제 — 의도된 구현 방식 (Intended Implementation Approach)

본 검토는 아래 구현 방식을 전제로 한다. (세부 설계는 DQ에서 확정)

| 영역 | 방식 |
|---|---|
| 플랫폼 | Electron (Chromium + Node.js), Windows 단독 PC |
| 데이터 저장 | 임베디드 DB(메타데이터·감사추적) + 로컬 디스크(이미지 파일) |
| 감사추적 | append-only 테이블 + 레코드 해시 체이닝(변조 탐지), 편집/삭제 UI·API 없음 |
| 원본 보호 | 이미지 + SHA-256 해시 저장, 읽기전용 속성, 읽을 때 해시 검증 |
| 시각 | OS 시스템 시계 사용 + 시간 변경 탐지·로깅 (네트워크 가능 시 NTP 보조) |
| 암호화 | 선택적 DB·이미지 암호화(encryption at rest), 키는 OS 보호 저장소(DPAPI 등) |
| 인증 | 로컬 계정, salted hash(argon2/bcrypt), RBAC, 세션 타임아웃 |
| 전자서명 | 재인증 + 서명 레코드(의미·시각) 기록 연결 |
| 설정 | 권한자 전용·감사추적되는 설정 프로필, 코어 플래그 잠금 |
| 백업 | 내장 스케줄 복사(경로 설정) 또는 비활성, 해시 무결성 점검 |
| LIMS | 커넥터 인터페이스(확장 지점), 기본 비활성 |

## 4. 일탈 등록부 및 예외 처리 (Deviation Register & Accepted Exceptions)

### 4.1 예외 처리 원칙 (Exception Principle)

- 아래 일탈은 모두 **단독 앱(SW)만으로는 해결 불가**하며, **회사마다 환경·설정이 달라** 제품이 일률적으로 떠안을 수 없다.
- 따라서 이들을 **"조건부 수용 예외(conditionally accepted exception)"**로 분류한다 — 제품의 결함이 아니라, 도입 환경에서 보완해야 할 항목으로 본다.
- 단, 각 예외에는 **필수 후속 조치(mandatory follow-up)**를 지정한다. 이는 **회사별 도입 시 반드시 수행·검증**되어야 한다.
- **필수 후속 조치가 이행·확인되지 않으면 해당 일탈은 미종결(open)** 상태이며, VP §7에 따라 **시스템을 운영에 사용할 수 없다.**
- 필수 후속 조치는 회사별 **도입 체크리스트(IQ)** 및 **SOP**로 전개되어 추적된다.

### 4.2 일탈/예외 등록부

| ID | 관련 URS | 일탈(예외) 내용 | 구분 | **필수 후속 조치 (반드시 수행)** | 검증 지점 |
|---|---|---|---|---|---|
| **DEV-01** | URS-041, 043 | 절대적 변조 **방지**가 아닌 변조 **탐지** 수준 | 환경 의존 예외 | OS 관리자 권한 제한 + 주기 무결성 점검 SOP 수립 | IQ(권한)·SOP |
| **DEV-02** | URS-044 | 신뢰 외부 시각원 보장 곤란 | 환경 의존 예외 | OS 시간변경 정책 제한, (가능 시) NTP 연결 | IQ(환경)·OQ(탐지) |
| **DEV-03** | URS-050 | 암호화 키 관리 제약 | 설계 의존 예외 | 회사 보안정책에 맞는 키 관리 방식 확정 | DQ·IQ |

### 4.3 필수 후속 조치 — 회사별 도입 체크리스트 (요약)

> 아래는 **각 회사 도입 시 IQ 단계에서 반드시 확인**하는 항목이다. 하나라도 미이행이면 해당 일탈 미종결.

- ☐ OS 관리자 권한 제한 적용 (DEV-01)
- ☐ OS 시스템 시간 변경 제한 정책 적용 / NTP 연결 (DEV-02)
- ☐ 암호화 키 관리 방식 회사 보안정책에 맞게 확정 (DEV-03)
- ☐ 백신·외부연결 차단·단독 PC 정책 적용 (DEV-04)
- ☐ 백업 복구 테스트 SOP 수립 (DEV-05)
- ☐ 장기 보존·매체 이관 SOP 수립 (DEV-06)
- ☐ (해당 시) LIMS 연동 변경관리 프로젝트 등록 (DEV-07)

## 5. 결론 (Conclusion)

- URS 42개 중 **충족(✔) 대부분**, 단독 앱이 떠안을 수 없는 항목은 **조건부 수용 예외 7건(DEV-01~07)**으로 처리.
- 모든 예외는 **소프트웨어 단독의 한계 + 회사별 환경 차이**에서 비롯되므로, 제품이 일률 구현하지 않고 **필수 후속 조치(환경·절차)**를 동반한 예외로 둔다 — RA의 "통제 3종류(SW·환경·절차)" 원칙과 일치.
- 각 예외의 필수 후속 조치는 **회사별 도입 시 IQ/SOP로 반드시 검증**되며, 미이행 시 일탈 미종결 → 운영 불가(게이트).
- 미해결(open) 일탈 없음(전 예외에 후속 조치·검증 지점 지정) → **DQ 단계 진입 가능**.
